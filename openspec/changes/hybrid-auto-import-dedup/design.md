## Context

当前 `jw` 自动导入流程会把浏览器历史按完整 URL 逐条写入本地存储，去重键为规范化后的完整 URL。该模型在高频内容站点会形成大量同域名近似页面记录，导致候选池噪声升高。

业务目标已明确：
- 采用 Hybrid 策略做自动导入整理。
- `jump` 默认落到最近深页面。
- 自动导入最终可以主导结果，但手动添加始终最高优先级。
- 相近深度仅保留一条代表页，不同深度不做强限制。

约束：
- 当前数据文件为 `~/.jw/store.json`，无数据库索引可用。
- 现有 HTTP/CLI 入口行为需要兼容，不应引入破坏性命令变更。
- 现有 URL 规范化仍需保留（协议安全、敏感参数脱敏）。

## Goals / Non-Goals

**Goals:**
- 为自动导入建立可解释、可回归的内置整理策略，显著降低同域名噪声。
- 在查询与跳转阶段明确来源优先级：manual > auto。
- 保留自动导入在无 manual 冲突时的主导能力（通过频次/新鲜度排序体现）。
- 保证不同深度页面不会被错误合并，满足“最近深页面”落点预期。

**Non-Goals:**
- 不引入浏览器插件或 Web 可视化面板。
- 不在本次变更中切换存储引擎（仍使用当前本地 JSON 存储）。
- 不在本次变更中实现复杂语义主题分类（仅使用轻量 topic 提取规则）。

## Decisions

### Decision 1: 为记录增加来源与整理元信息（source-aware entries）
- Choice:
  - 在现有 entry 结构中补充来源与整理字段：`source`、`group_key`、`depth_bucket`、`topic_key`、`representative`（或等价语义）。
- Rationale:
  - 现有去重与排序仅基于完整 URL，无法表达“自动导入整理后的代表关系”。
  - 将元信息内联到 entry，便于在加载、导入、排序路径直接使用。
- Alternatives considered:
  - 新建独立索引文件维护聚合关系：读取路径复杂，状态一致性成本高。
  - 仅在内存态计算聚合，不落盘：重启后策略不可复现，难以调试与回归。

### Decision 2: 自动导入分组键采用 `host + depth-bucket + topic-key`
- Choice:
  - 对每条自动导入 URL 抽取：
    - `host`（规范化后的主机）
    - `depth-bucket`（路径深度分桶）
    - `topic-key`（轻量路径主题键）
  - 组合得到 `group_key`，用于“相近深度仅保留一条”的归并边界。
- Rationale:
  - 单纯按 host 会过度合并，丢失深页面可达性。
  - 将深度纳入键可满足“不同深度不限制”。
  - topic 维度可避免同深度不同内容流被错误互相替换。
- Alternatives considered:
  - 仅 host+depth：在 docs/news/forum 混合站点仍会互相覆盖。
  - host+full-path-prefix：分组过细，降噪收益不足。

### Decision 3: 自动导入在组内只保留一个“代表页”
- Choice:
  - 对同一 `group_key` 的 auto 记录，仅保留一个代表页；新记录进入时执行代表页替换策略（默认最近优先，结合稳定性约束）。
- Rationale:
  - 这是“相近深度保留1条”的直接落地。
  - 通过组内替换替代无限增长，限制噪声与存储膨胀。
- Alternatives considered:
  - 每组保留 Top-N：复杂度更高且会重新引入噪声。
  - 完全只保留域名主页：违背“默认最近深页面”目标。

### Decision 4: 排序采用来源优先级乘子 + 行为分
- Choice:
  - 总分由来源优先级乘子与现有行为分（频次+时间衰减+关键词匹配）组合。
  - manual 乘子高于 auto；当无 manual 强匹配时，auto 仍可凭行为分主导结果。
- Rationale:
  - 与产品要求一致：手动最高优先级，但自动导入仍是主要学习来源。
- Alternatives considered:
  - 绝对优先（只要有 manual 就总是第一）：会让自动导入长期失效。
  - 完全统一权重：无法体现手动意图的确定性价值。

### Decision 5: `jump` 默认落到最近深页面
- Choice:
  - 在候选决策中优先选择满足“深页面条件”的最新代表记录作为默认落点。
- Rationale:
  - 贴合用户行为预期，减少每次跳转先回首页再二次导航的成本。
- Alternatives considered:
  - 固定跳首页：最稳但效率低，不符合当前目标。
  - 固定最高频页面：会偏向历史旧热点，弱化“最近”语义。

### Decision 6: 旧数据采用兼容来源 `legacy`
- Choice:
  - 既有 `store.json` 记录在迁移阶段标记为 `legacy`，使用中性优先级，不强制当作 manual 或 auto。
- Rationale:
  - 现有数据来源不可追溯，强行归类会造成排序偏差或行为突变。
- Alternatives considered:
  - 全部当 manual：可能人为抬升历史噪声。
  - 全部当 auto：可能错误压低用户曾手动强化的目标。

## Risks / Trade-offs

- [分组键过粗导致误合并] → 通过 `topic-key` 参与分组并增加回归样本验证（docs/blog/search 等站型）。
- [分组键过细导致降噪不明显] → 增加观测指标（每域记录数、候选重复率），必要时调整 depth/topic 规则。
- [来源权重设置不当导致跳转回退] → 以基准集做 A/B 回放，确保手动命中优先率与 top1 命中率不下降。
- [JSON 存储元信息增长带来 I/O 压力] → 控制字段规模、减少冗余，评估后续是否迁移到结构化存储。
- [legacy 兼容期行为不稳定] → 保留可回滚策略与一次性重建入口（必要时重算自动导入池）。

## Migration Plan

1. 定义并引入新 entry 元字段与默认值（包含 `legacy` 兼容语义）。
2. 在加载阶段对旧记录做无损兼容映射，不破坏既有数据可读性。
3. 在自动导入路径启用新分组与代表页替换策略（仅对 `source=auto` 生效）。
4. 在查询/跳转路径启用来源优先级与“最近深页面”决策。
5. 通过回归数据集验证：
   - 手动优先命中率
   - 同域噪声下降幅度
   - 跳转成功率与延迟
6. 逐步发布；若指标异常，回滚到旧排序策略并保留兼容读取。

## Open Questions

- `depth-bucket` 的分桶边界采用固定阈值还是站点自适应？
- `topic-key` 的抽取规则是否只基于首段 path，还是允许二级段参与？
- “最近深页面”的深度下限是否统一定义为 `depth >= 2`，或按域类型可配置？
- 是否需要公开 `jw config` 开关用于切换“严格降噪/宽松降噪”模式？
